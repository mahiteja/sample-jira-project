EPIC-1: Customer Onboarding & Authentication (100 SP)

Description: Enable secure customer registration and identity verification compliant with KYC/AML regulations. Implement multi-factor and biometric authentication for enhanced security.

Business Value: Foundation for all customer interactions. Ensures regulatory compliance and secure access to banking services.

KYC AML security phase-1 mvp

üìò User Story: User Registration (8 SP)

As a new customer
I want to register for online banking
So that I can access my accounts digitally and manage my finances online

High Priority KYC phase-1

Acceptance Criteria:

Prerequisites:

  • Customer must have valid email address and personal information
  • Registration page must be accessible
  • System must be operational

Scenario: New customer registers successfully

  1. New user visits registration page
  2. User provides valid personal information (name, email, phone, date of birth)
  3. User creates strong password meeting security requirements (min 8 chars, uppercase, lowercase, number, special char)
  4. User submits registration form
  5. System validates all information
  6. System sends verification email
  7. User clicks verification link in email
  8. Account is created with "pending" status
  9. User receives welcome email
  10. KYC verification process is automatically initiated

Security Requirements:

  • Password hashing with bcrypt (salt rounds: 12)
  • Email verification required before account activation
  • CAPTCHA to prevent bot registrations
  • Rate limiting: max 5 registration attempts per IP per hour

Compliance Requirements:

  • KYC: Collect minimum required information (name, DOB, address, SSN)
  • GDPR: User consent for data processing
  • Audit: Log all registration attempts with timestamp, IP address

üìò User Story: KYC Document Upload (13 SP)

As a new customer
I want to upload my identity documents
So that I can complete the KYC verification process and activate my account

Critical Priority KYC compliance

Acceptance Criteria:

Prerequisites:

  • Customer must have registered account
  • Customer must have valid government-issued ID
  • Customer must have proof of address document (dated within last 3 months)

Scenario: Customer uploads identity documents successfully

  1. Customer logs into account with "pending" status
  2. Customer navigates to KYC document upload section
  3. Customer selects document type (government ID)
  4. Customer uploads clear photo or scan of ID (max 5MB, PDF/JPG/PNG)
  5. Customer selects proof of address document type
  6. Customer uploads proof of address (utility bill, bank statement, etc.)
  7. All documents are encrypted with AES-256 during upload
  8. System stores documents securely in encrypted storage
  9. OCR automatically extracts information from documents
  10. Compliance team is notified for manual review
  11. Customer sees upload confirmation with reference number
  12. Customer receives email confirmation that documents are under review

Technical Requirements:

  • File upload: Max 5MB per document
  • Supported formats: PDF, JPG, JPEG, PNG
  • OCR service integration for data extraction
  • Document storage: encrypted S3 bucket with restricted access

üìò User Story: Transfer Between Own Accounts (13 SP)

As a banking customer
I want to transfer money between my own accounts
So that I can manage my finances efficiently

Critical Priority transfers core-banking

Acceptance Criteria:

Prerequisites:

  • Customer must have at least two active accounts
  • Source account must have sufficient balance
  • Customer must be logged in with MFA

Scenario: Successful transfer between own accounts

  1. Customer has checking account with balance $5000 and savings with $1000
  2. Customer navigates to "Transfer Funds" page
  3. Customer selects checking account (ending in "1234") as source
  4. Customer selects savings account (ending in "5678") as destination
  5. Customer enters transfer amount "$500"
  6. Customer optionally adds description "Monthly savings"
  7. System validates sufficient balance exists
  8. Customer clicks "Continue" button
  9. System displays transfer preview with all details
  10. Customer reviews and clicks "Confirm Transfer"
  11. If amount exceeds $500, MFA verification prompt appears
  12. Customer completes MFA verification
  13. System processes transfer immediately in a database transaction
  14. Checking account balance updates to $4500
  15. Savings account balance updates to $1500
  16. Confirmation message displays "Transfer successful"
  17. Transaction appears in history of both accounts immediately
  18. Confirmation email is sent within 30 seconds
  19. Confirmation SMS is sent within 30 seconds

Scenario: Transfer with insufficient funds

  1. Customer has checking account with balance $100
  2. Customer attempts to transfer $500 from checking
  3. System validates available balance
  4. System displays error message "Insufficient funds"
  5. Transfer is not processed
  6. Account balances remain unchanged
  7. No notifications are sent

Technical Requirements:

  • API Endpoint: POST /api/v1/transfers/internal
  • Response Time: < 500ms for 95th percentile
  • Database: ACID transaction to ensure atomicity
  • Concurrency: Row-level locking for simultaneous transfers
  • Idempotency: Support idempotent requests to prevent duplicates

üìò User Story: Real-time Fraud Detection (21 SP)

As the banking system
I need to detect suspicious transactions in real-time
So that I can prevent fraudulent activities and protect customers

Critical Priority fraud-detection ml

Acceptance Criteria:

Prerequisites:

  • Fraud detection engine must be active and monitoring
  • ML models must be trained on fraud patterns

Scenario: Detect suspicious transaction

  1. Fraud detection system monitors all transactions in real-time
  2. Customer attempts transaction (e.g., $2000 purchase from new location)
  3. System analyzes transaction against customer's historical patterns
  4. System detects unusual characteristics:
    • Transaction from location 500 miles from usual
    • Amount 10x higher than average
    • New merchant/category
  5. Risk score is calculated (e.g., 85/100 - high risk)
  6. Transaction is automatically held/blocked before processing
  7. Customer immediately receives SMS alert: "Unusual transaction detected. Was this you?"
  8. Customer receives push notification with transaction details
  9. Notification includes "Verify" or "Report Fraud" buttons
  10. If customer verifies within 10 minutes, transaction proceeds
  11. If customer reports fraud, transaction is permanently blocked
  12. If no response within 10 minutes, transaction is blocked
  13. Security team is notified for manual review
  14. All fraud events are logged with timestamp, risk score, outcome
